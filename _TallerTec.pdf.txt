1. RESUMEN EJECUTIVO 
El presente documento constituye la propuesta formal para el desarrollo de TallerTec, un 
sistema integral de gestión de talleres deportivos diseñado para resolver una 
problemática real y documentada que afecta a la totalidad de los estudiantes del 
Tecnológico Nacional de México Campus Matehuala. El proyecto surge de la experiencia 
directa del equipo de desarrollo como usuarios y administradores del sistema actual, lo 
que garantiza un entendimiento profundo del problema y sus implicaciones. 
Actualmente, el proceso de inscripción, control de asistencia y acreditación de las 20 
horas de talleres deportivos requeridas para titulación opera mediante un sistema 
completamente manual: ferias presenciales obligatorias, hojas de registro en papel, 
transcripción manual a Excel, y generación lenta de constancias. Este proceso genera 
errores frecuentes, pérdida de información, frustración estudiantil, y una carga 
administrativa innecesaria para el personal de la oficina de deportes. 
TallerTec propone una solución tecnológica completa que digitaliza todo el flujo: 
inscripción en línea sin necesidad de asistir presencialmente, registro de asistencia 
mediante códigos QR, control de horas en tiempo real visible para estudiantes, y 
generación automática de constancias. El sistema contempla tres tipos de usuarios con 
interfaces diferenciadas: estudiantes (app/PWA), administradores de taller (panel de 
gestión), y personal de oficina (dashboard administrativo). 
El proyecto se desarrollará en 16 semanas utilizando metodología Scrum, con un stack 
tecnológico moderno: Next.js 14 para el frontend web, React Native/Expo para la 
aplicación móvil, y Supabase como backend (PostgreSQL + Auth + Storage). El costo de 
desarrollo es $0 MXN aprovechando los tiers gratuitos de las plataformas seleccionadas. 
El resultado será un sistema funcional listo para implementación piloto en el campus, con 
potencial de escalamiento a otros campus del TecNM.

2. IDENTIFICACIÓN DEL PROBLEMA 
2.1 Contexto Institucional 
El Tecnológico Nacional de México (TecNM) establece como requisito de titulación que 
todos los estudiantes de nivel licenciatura completen 20 horas de actividades deportivas 
o culturales a través de los talleres institucionales. Este requisito aplica a los más de 
5,000 estudiantes activos del campus Matehuala y se replica en los 254 campus del 
TecNM a nivel nacional. 
Los talleres deportivos disponibles incluyen disciplinas como fútbol, voleibol, básquetbol, 
ajedrez, atletismo, entre otros. Cada taller es coordinado por un responsable (que puede 
ser un docente o un estudiante avanzado) quien debe llevar el control de asistencia y 
reportar las horas acumuladas de cada participante a la oficina de deportes. 
El proceso actual fue diseñado hace años para un volumen menor de estudiantes y no 
ha evolucionado con las necesidades actuales. La falta de digitalización genera cuellos 
de botella, errores, y una experiencia frustrante tanto para estudiantes como para el 
personal administrativo. 
2.2 Descripción del Proceso Actual (AS-IS) 
FASE 1 - Inscripción (Feria de Talleres): Dos veces al año, la institución organiza una 
'Feria de Talleres Deportivos' en el auditorio. Todos los estudiantes (nuevos y regulares) 
deben asistir presencialmente en un horario específico de 1-2 días. Se les entregan hojas 
impresas donde deben escribir manualmente: número de control, nombre completo, 
carrera, semestre, teléfono, y taller deseado. Estas hojas se entregan a los encargados 
de cada taller. 
FASE 2 - Captura de Datos: El personal de la oficina de deportes recibe las hojas de 
todos los talleres y debe transcribir manualmente cada registro a una hoja de cálculo 
Excel. Con cientos de estudiantes y escritura a mano frecuentemente ilegible, este 
proceso toma varios días y genera errores de transcripción (nombres mal escritos, 
números de control incorrectos, duplicados, omisiones). 
FASE 3 - Control de Asistencia: Cada responsable de taller lleva su propio control de 
asistencia, generalmente en una libreta o lista impresa. No existe un formato 
estandarizado. Algunos responsables son rigurosos, otros no. Las horas se calculan de 
manera aproximada. 
FASE 4 - Reporte de Horas: Al final del semestre, los responsables de taller deben 
reportar las horas de cada estudiante a la oficina. Este reporte se hace de manera

2.3 Análisis de Causa Raíz (Diagrama de Ishikawa) 
El problema central identificado es: 'Proceso ineficiente, propenso a errores y frustrante 
para la gestión de talleres deportivos'. Las causas se agrupan en las siguientes 
categorías: 
2.3.1 Métodos y Procesos 
El proceso de inscripción requiere presencia física obligatoria, excluyendo a estudiantes 
con conflictos de horario, trabajo, o situaciones personales. No existe un procedimiento 
estandarizado para el control de asistencia entre los diferentes talleres. La comunicación 
entre responsables de taller y oficina es informal y no tiene trazabilidad. No hay un 
proceso definido para altas tardías (estudiantes que se unen después de la feria). 
2.3.2 Personas 
Los responsables de taller (especialmente estudiantes) no tienen capacitación formal en 
gestión administrativa. La carga de trabajo del personal de oficina se concentra en 
períodos pico (inicio de semestre, fin de semestre, período de titulación). Los errores de 
transcripción son inevitables con el volumen actual de datos manuales. La rotación de 
responsables de taller genera pérdida de información histórica. 
2.3.3 Tecnología 
El único 'sistema' es un archivo Excel que no tiene validaciones, respaldos automáticos, 
ni control de acceso. No existe una base de datos centralizada ni sistema de información. 
Los estudiantes no tienen forma de consultar sus horas acumuladas sin ir 
presencialmente a preguntar. No hay automatización en ninguna parte del proceso. 
2.3.4 Información 
Los datos capturados en papel son frecuentemente ilegibles o incompletos. No existe 
historial de asistencias individuales, solo totales aproximados. La información está 
fragmentada entre múltiples libretas de diferentes talleres. No hay reportes ni estadísticas 
disponibles para la toma de decisiones.

2.5 Impacto Cuantificable del Problema 
Tiempo perdido por estudiantes en inscripción presencial: Estimando 1,500 estudiantes 
en la feria, con tiempo promedio de 45 minutos (traslado + espera + registro), se pierden 
1,125 horas-estudiante por semestre. Si el valor del tiempo de un estudiante se estima 
conservadoramente en $50 MXN/hora, el costo de oportunidad es de $56,250 MXN por 
semestre solo en inscripciones. 
Tiempo del personal en transcripción manual: Con 1,500 registros y un tiempo promedio 
de 2 minutos por registro (lectura de escritura a mano, corrección de errores, captura), 
se requieren aproximadamente 50 horas de trabajo administrativo por semestre solo para 
captura inicial. 
Errores de datos: Se estima conservadoramente un 5-10% de errores en la transcripción 
(nombres mal escritos, números de control incorrectos). Con 1,500 registros, esto 
significa 75-150 estudiantes con información incorrecta que genera re-trabajo y 
frustración. 
Consultas presenciales: Si el 30% de los estudiantes (450) acude al menos una vez por 
semestre a preguntar por sus horas, con un tiempo promedio de 10 minutos por consulta, 
se consumen 75 horas de atención que podrían evitarse con un sistema de consulta en 
línea.

3. PROPUESTA DE SOLUCIÓN 
3.1 Descripción General del Sistema 
TallerTec es un sistema integral de gestión que digitaliza completamente el ciclo de vida 
de los talleres deportivos institucionales, desde la inscripción hasta la generación de 
constancias. El sistema se compone de tres interfaces principales, cada una diseñada 
para las necesidades específicas de su tipo de usuario: 
3.1.1 Aplicación para Estudiantes (TallerTec App) 
Progressive Web App (PWA) instalable en cualquier dispositivo que permite a los 
estudiantes: explorar el catálogo de talleres disponibles con horarios, ubicaciones y 
cupos; inscribirse en línea a uno o más talleres sin necesidad de asistencia presencial; 
visualizar su código QR personal único para registro de asistencia; consultar en tiempo 
real las horas acumuladas por taller y el total; ver el historial detallado de asistencias 
registradas; solicitar su constancia de créditos cuando completen las 20 horas; y recibir 
notificaciones sobre horarios, eventos y recordatorios. 
3.1.2 Panel para Responsables de Taller (TallerTec Admin) 
Interfaz web responsive que permite a los coordinadores de cada taller: registrar 
asistencia escaneando el código QR de los estudiantes con la cámara del celular; ver la 
lista completa de estudiantes inscritos en su taller; consultar el historial de asistencias de 
cada estudiante; agregar estudiantes que se inscriben fuera del período regular; reportar 
incidencias o situaciones especiales; y visualizar estadísticas de su taller (asistencia 
promedio, estudiantes activos, etc.). 
3.1.3 Dashboard para Oficina de Deportes (TallerTec Dashboard) 
Panel administrativo completo que permite al personal de la oficina: gestionar el catálogo 
de talleres (crear, editar, desactivar); administrar responsables de taller y sus permisos; 
ver el estado global de todos los talleres en tiempo real; validar y aprobar horas 
reportadas por los responsables; generar constancias de créditos de manera automática 
con un clic; exportar datos a Excel para reportes institucionales; consultar estadísticas y 
métricas del programa deportivo; y gestionar períodos de inscripción y configuraciones 
del sistema.

procesos manuales actuales, reduzca el tiempo de inscripción de 45 minutos 
presenciales a menos de 5 minutos en línea, proporcione visibilidad en tiempo real de

1. Digitalización de Inscripciones (Specific, Measurable): Permitir que el 100% de las

5. ARQUITECTURA PROPUESTA 
5.1 Arquitectura General 
El sistema sigue una arquitectura de tres capas con separación clara de 
responsabilidades, optimizada para desarrollo ágil con un equipo pequeño: 
CAPA DE PRESENTACIÓN: Tres interfaces de usuario desarrolladas con Next.js 14 y 
Tailwind CSS. La PWA para estudiantes es una aplicación web progresiva instalable con 
manifest.json y service worker para funcionalidad offline básica. El panel para 
responsables de taller es una interfaz web responsive optimizada para uso en 
dispositivos móviles. El dashboard administrativo es una interfaz web completa con 
componentes de visualización de datos. 
CAPA DE LÓGICA DE NEGOCIO: API Routes de Next.js que implementan la lógica del 
sistema. Endpoints RESTful para todas las operaciones CRUD. Validación de datos con 
Zod. Autenticación y autorización con Supabase Auth y middleware personalizado. 
Generación de PDFs con React-PDF o jsPDF. 
CAPA DE DATOS: Supabase como Backend-as-a-Service. PostgreSQL como base de 
datos relacional con Row Level Security (RLS) para seguridad a nivel de fila. Supabase 
Auth para gestión de usuarios y sesiones. Supabase Storage para almacenamiento de 
archivos (fotos de perfil, constancias generadas).

5.3 Flujos Principales del Sistema 
Flujo de Inscripción: Estudiante accede a la PWA → Inicia sesión con correo institucional 
→ Ve catálogo de talleres disponibles → Selecciona taller y confirma inscripción → 
Sistema valida cupo disponible → Crea registro en tabla inscripciones → Genera código 
QR único si es primera inscripción → Envía confirmación por email → Estudiante puede 
ver su inscripción en 'Mis Talleres'. 
Flujo de Registro de Asistencia: Estudiante llega al taller → Muestra su código QR en la 
app → Responsable abre panel y selecciona 'Registrar Asistencia' → Escanea QR con 
cámara del celular → Sistema valida que el estudiante está inscrito en ese taller → 
Registra hora de entrada → Al finalizar sesión, responsable puede registrar hora de 
salida o usar duración estándar → Sistema calcula horas y actualiza acumulado.

10. DOCUMENTACIÓN Y ESTÁNDARES 
10.1 Estándares de Código 
JavaScript/TypeScript: Se utilizará TypeScript para todo el código del proyecto, 
aprovechando el tipado estático para prevenir errores. ESLint con configuración de 
Next.js para linting. Prettier para formateo automático. Convención de nombres: 
camelCase para variables y funciones, PascalCase para componentes React, 
UPPER_SNAKE_CASE para constantes.

reutilizables en /components. Hooks personalizados en /hooks. Utilidades en /lib. Tipos 
TypeScript en /types. 
Base de Datos: Nombres de tablas en snake_case y plural (usuarios, talleres). Nombres 
de columnas en snake_case. Uso de UUIDs como primary keys. Timestamps created_at 
y updated_at en todas las tablas. 
10.2 Control de Versiones 
Repositorio: GitHub con repositorio privado compartido con el profesor como 
colaborador. 
Estrategia de Branching: Git Flow simplificado. Rama 'main' siempre desplegable. Rama 
'develop' para integración. Features en ramas 'feature/nombre-descriptivo'. Hotfixes en 
ramas 'hotfix/descripcion'. Pull Requests obligatorios para merge a develop y main. 
Commits: Conventional Commits. Formato: tipo(alcance): descripción. Tipos: feat, fix, 
docs, style, refactor, test, chore. Ejemplos: 'feat(auth): implementar registro con email 
institucional', 'fix(qr): corregir lectura en cámaras antiguas'.
